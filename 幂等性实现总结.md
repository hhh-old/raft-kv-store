# 幂等性实现总结

## 一、实现概述

已成功为 Raft KV Store 实现基于 RequestId 的幂等性去重机制。

### 核心思想

1. **客户端生成 requestId**：每个业务操作生成唯一的 requestId
2. **服务端缓存处理结果**：使用 ConcurrentHashMap 缓存已处理请求的结果
3. **重复请求直接返回**：检测到相同 requestId 时，直接返回缓存结果，不重复执行

### 2.1 KVStoreStateMachine.java

**新增字段**：

```java
// 已处理请求的缓存（用于幂等去重）
private final Map<String, KVResponse> processedRequests = new ConcurrentHashMap<>();

// 请求缓存的最大大小（防止内存溢出）

```

**新增方法**：
```java
// 检查是否是重复请求
public boolean isDuplicateRequest(String requestId)

// 缓存操作结果
private void cacheResult(KVTask task)

// 获取已缓存的请求结果
public KVResponse getCachedResult(String requestId)
```

**修改方法**：
```java
// onApply 方法中添加幂等检查
public void onApply(Iterator iterator) {
    while (iterator.hasNext()) {
        KVTask task = deserialize(data);
        if (task != null) {
            // 幂等检查：如果 requestId 已处理，跳过执行
            if (isDuplicateRequest(task.getRequestId())) {
                LOG.debug("Duplicate request skipped: {}", task.getRequestId());
            } else {
                applyTask(task);
                cacheResult(task);  // 缓存结果
            }
        }
        iterator.next();
    }
}
```

**幂等检查逻辑**：
```java
public KVResponse put(String key, String value, String requestId) {
    // 如果没有提供 requestId，生成一个（向后兼容）
    if (requestId == null || requestId.isEmpty()) {
        requestId = UUID.randomUUID().toString();
    }

    // 幂等检查：如果是重复请求，直接返回缓存的结果
    KVResponse cachedResult = stateMachine.getCachedResult(requestId);
    if (cachedResult != null) {
        LOG.info("Request already processed: {}", requestId);
        cachedResult.setServedBy(getCurrentEndpoint());
        return cachedResult;
    }

    // 正常执行...
}
```

## 三、API 使用方式

### 3.1 PUT 操作

**方式 1：不提供 requestId（向后兼容）**
```bash
curl -X PUT http://localhost:9081/kv/name \
  -H "Content-Type: application/json" \
  -d '{"value": "Alice"}'
```

**方式 2：提供 requestId（支持幂等）**
```bash
curl -X PUT http://localhost:9081/kv/name \
  -H "Content-Type: application/json" \
  -d '{
    "value": "Alice",
    "requestId": "req-001"
  }'
```

### 3.2 DELETE 操作

**方式 1：不提供 requestId**
```bash
curl -X DELETE http://localhost:9081/kv/name
```

**方式 2：提供 requestId**
```bash
curl -X DELETE "http://localhost:9081/kv/name?requestId=del-001"
```

## 四、工作流程

### 4.1 首次请求

```
客户端                 服务端
  |                      |
  |--- PUT (reqId=001)-->|
  |                      |-- 检查缓存：未找到
  |                      |-- 写入 Raft 日志
  |                      |-- 应用到状态机
  |                      |-- 缓存结果 (reqId=001 -> response)
  |<-- 成功响应 ----------|
```

### 4.2 重复请求（幂等）

```
客户端                 服务端
  |                      |
  |--- PUT (reqId=001)-->|
  |                      |-- 检查缓存：找到！
  |                      |-- 直接返回缓存结果
  |<-- 缓存响应 ----------|
```

### 4.3 状态机层面

```
onApply(iterator):
  while iterator.hasNext():
    task = deserialize(data)
    
    if isDuplicateRequest(task.requestId):
      // 跳过执行（幂等）
      continue
    else:
      // 执行操作
      applyTask(task)
      
      // 缓存结果
      cacheResult(task)
```

## 五、关键特性

### 5.1 线程安全

- 使用 `ConcurrentHashMap` 存储请求缓存
- `onApply` 方法由 Raft 保证单线程调用
- 读写操作都是线程安全的

### 5.2 内存保护

```java
// 限制缓存大小
private static final int MAX_CACHE_SIZE = 10000;

if (processedRequests.size() > MAX_CACHE_SIZE) {
    LOG.warn("Request cache size exceeded, clearing cache");
    processedRequests.clear();
}
```

### 5.3 向后兼容

## 八、局限性与改进方向

### 8.1 当前局限

1. **节点重启后缓存丢失**
   - 问题：重启后 processedRequests 为空
   - 影响：无法去重重启前的请求
   - 解决：持久化 requestId 到磁盘

2. **缓存清理策略简单**
   - 问题：超过 MAX_CACHE_SIZE 后全部清空
   - 影响：可能丢失未过期的 requestId
   - 解决：使用 LRU 淘汰策略

3. **不支持跨节点去重**
   - 问题：每个节点的缓存独立
   - 影响：Leader 切换后可能重复执行
   - 解决：共享缓存或使用 Raft 日志去重

### 8.2 改进建议

**优先级 1：持久化 requestId**
```java
// 在 saveData() 时同时保存 requestId
private void saveData() {
    // 保存 kvStore
    objectMapper.writeValue(kvStoreFile, kvStore);
    
    // 保存 processedRequests
    objectMapper.writeValue(requestIdFile, processedRequests);
}
```

**优先级 2：LRU 缓存**
```java
// 使用 LinkedHashMap 实现 LRU
private final Map<String, KVResponse> processedRequests = 
    new LinkedHashMap<String, KVResponse>(16, 0.75f, true) {
        @Override
        protected boolean removeEldestEntry(Map.Entry eldest) {
            return size() > MAX_CACHE_SIZE;
        }
    };
```

**优先级 3：基于 Raft 日志去重**

## 九、总结

### 9.1 已实现功能

✅ 基于 RequestId 的幂等性去重机制
✅ PUT 和 DELETE 操作支持幂等
✅ 向后兼容（不提供 requestId 时自动生成）
✅ 线程安全的缓存实现
✅ 内存保护机制（MAX_CACHE_SIZE）
✅ 完整的测试用例

### 9.2 使用建议

1. **客户端必须生成 requestId**：基于业务逻辑生成（如订单号、用户ID等）
2. **重试时使用相同 requestId**：保证幂等性
3. **requestId 必须唯一**：不同业务操作使用不同 requestId
4. **监控缓存命中率**：观察日志中的 "Request already processed" 频率

### 9.3 客户端示例

```java
// 推荐：基于业务键生成 requestId
String requestId = "put-" + key + "-" + System.currentTimeMillis();

// 重试循环
for (int i = 0; i < 3; i++) {
    KVResponse response = httpClient.put("/kv/" + key, 
        Map.of("value", value, "requestId", requestId));
    
    if (response.isSuccess()) {
        return response;  // 成功
    }
    
    // 超时或其他错误，重试（使用相同 requestId）
}
```

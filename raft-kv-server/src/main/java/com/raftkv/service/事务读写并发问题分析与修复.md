# Raft KV Store 事务读写并发问题分析与修复

## 一、问题描述

事务操作（Txn）中包含多个写语句（PUT/DELETE），这些写语句在状态机的 `onApply` 方法中是一条一条逐条执行的。问题是：如果事务写入的中间过程中来了一个读请求，是否会读到事务的"中间结果"？

**答案：在修复之前，确实会读到中间结果。**

### 根本原因

1. **Raft `onApply` 的串行粒度 ≠ 事务操作粒度**
   - Raft 保证 `onApply` 单线程串行执行，但串行化的最小单位是**日志条目（Log Entry）**
   - 一个事务（Txn）就是一个日志条目，但条目内部的多个 PUT/DELETE 操作是逐条执行的
   - 每次 `putWithRevision()` 都会立即写入 `ConcurrentSkipListMap`，对其他并发线程立即可见

2. **ReadIndex 与 `onApply` 存在时间窗口**
   - `node.readIndex()` 记录 `commitIndex` → 发心跳确认 Leader → 等待 `lastApplied >= commitIndex` → 回调读取
   - 回调触发的条件只是"某个日志索引已 apply"，而不是"所有日志已 apply"
   - 如果在事务 `onApply` 执行到第 N 条写入时，ReadIndex 条件已满足，回调就会读取，此时只能看到前 N 条写入的结果

3. **MVCC `getLatest()` 读取最新版本**
   - 之前读操作使用 `mvccStore.getLatest(key)`，返回该 key 的最新版本
   - 事务逐条写入后，最新版本就是刚写入的那条，所以读操作能看到中间状态

---

## 二、修复方案：MVCC 快照读（Revision-based Snapshot Read）

### 核心思路

> **不是让写操作对外不可见，而是让读操作基于固定 revision 做快照读取。**

在发起 `node.readIndex()` 之前，先采样当前的 `currentRevision`。这个 revision 代表了 ReadIndex 发起时刻状态机的快照点。回调中使用 `getAtRevision(key, readRevision)` 而不是 `getLatest()`，这样事务逐条写入的任何新 revision 都大于采样值，对此读操作完全透明。

### 代码修改

#### 1. KVStoreStateMachine 暴露快照读方法

```java
public MVCCStore.KeyValue getAtRevision(String key, long revision) {
    return mvccStore.getAtRevision(key, revision);
}

public Map<String, MVCCStore.KeyValue> getAllAtRevision(long revision) {
    return mvccStore.getAllAtRevision(revision);
}
```

#### 2. RaftKVService.get() 基于采样 revision 读取

```java
public KVResponse get(String key) {
    // 在 ReadIndex 发起前先采样当前 revision
    long readRevision = stateMachine.getCurrentRevision();

    // 将 key:revision 编码到 requestContext，传递到回调
    byte[] requestContext = (key + "\0" + readRevision).getBytes(StandardCharsets.UTF_8);

    node.readIndex(requestContext, new ReadIndexClosure() {
        @Override
        public void run(Status status, long index, byte[] reqCtx) {
            // 解析 key 和采样 revision
            String ctxStr = new String(reqCtx, StandardCharsets.UTF_8);
            int sep = ctxStr.indexOf('\0');
            String readKey = ctxStr.substring(0, sep);
            long rev = Long.parseLong(ctxStr.substring(sep + 1));

            // 基于固定 revision 做快照读
            MVCCStore.KeyValue kv = stateMachine.getAtRevision(readKey, rev);
            // ...
        }
    });
}
```

`getAll()` 同理，采样 revision 后使用 `getAllAtRevision(rev)`。

## 三、方案正确性：三种时序场景

### 场景一：读发起时，事务 `generateRevision` 早于读采样

```
时间轴：
  T1: 写事务开始 → generateRevision() → currentRevision: 10 → 11
  T2: 写事务在 onApply 中逐条写入（单线程，不可中断）
  T3: 读请求到来 → 采样 readRevision = 11 → 发起 ReadIndex
  T4: ReadIndex 等待 lastApplied >= commitIndex
      （此 commitIndex 是事务日志的索引，onApply 完成后 lastApplied 才递增）
  T5: onApply 全部完成 → lastApplied 追上 → ReadIndex 回调触发
      → getAtRevision(key, 11)
      → 此时 rev=11 的所有数据已完整写入
      → 读到完整的事务结果 ✓
```

**为什么正确**：ReadIndex 回调触发的前提是 `lastApplied` 追上 `commitIndex`，而 `lastApplied` 只在整个日志条目 apply 完毕后才递增。所以回调触发时事务已完整写入，不会看到中间状态。

---

### 场景二：读发起时，事务尚未开始（读采样早于 `generateRevision`）

```
时间轴：
  T1: 读请求到来 → 采样 readRevision = 10 → 发起 ReadIndex
  T2: 写事务开始 → generateRevision() → currentRevision: 10 → 11
  T3: 写事务逐条写入 rev=11 的中间状态
  T4: ReadIndex 回调触发
      → getAtRevision(key, 10)
      → floorEntry(rev ≤ 10)，rev=11 完全不可见
      → 读到事务提交前的完整快照 ✓
```

**为什么正确**：`readRevision=10 < 事务写入的 rev=11`，MVCC `floorEntry` 天然隔离，看不到任何事务数据。

---

### 场景三：读采样恰好在 `generateRevision` 之后、第一条写入之前

```
时间轴：
  T1: 写事务调用 generateRevision() → currentRevision: 10 → 11
  T2: 读请求采样 readRevision = 11 → 发起 ReadIndex
      （此时 rev=11 的数据还一条都没写！）
  T3: 写事务在 onApply 中逐条写入 key1(rev=11), key2(rev=11)...
  T4: ReadIndex 等待 lastApplied >= commitIndex
      （关键：此 commitIndex 就是当前正在 apply 的这个事务日志！）
  T5: onApply 完成 → lastApplied 递增 → ReadIndex 回调触发
      → getAtRevision(key, 11)
      → 此时所有 rev=11 的数据已完整写入
      → 读到完整的事务结果 ✓
```

**为什么正确**：Raft `onApply` 是单线程的，一个日志条目内的所有操作要么全部完成（`lastApplied` 递增），要么都没完成。不存在"日志条目 apply 到一半但 lastApplied 已递增"的情况。ReadIndex 的 `commitIndex` 等待天然保证了回调时事务已完整 apply。

---

## 四、总结

| 场景 | 采样 readRevision | 事务 mainRev | ReadIndex 回调时状态 | 结论 |
|------|-------------------|-------------|---------------------|------|
| 读先于事务 | N | N+1 | 事务未开始 | 读快照 N，事务不可见，隔离 ✓ |
| 读采样在事务 `generateRevision` 之后 | N+1 | N+1 | 等 `onApply` 完整执行后回调 | 读到完整事务 ✓ |
| 读采样在 `generateRevision` 之前 | N | N+1 | 事务可能已开始 | 读快照 N，天然隔离 ✓ |

### 三个关键保证

1. **ReadIndex 的 apply 等待**：回调触发时当前日志条目已完整 apply，不存在中间状态
2. **MVCC 快照读**：`readRevision` 在发起时采样固定，事务写入的 revision > `readRevision` 则完全不可见
3. **Raft `onApply` 单线程串行**：`generateRevision()` 和所有写入在同一个线程中完成，没有并发交叉

### 修复后的正确性

- **事务的原子可见性**：要么读到事务完整提交后的状态（rev = N+1），要么读到事务开始前的快照（rev ≤ N），永远不会读到中间状态。
- **线性一致性**：ReadIndex 机制保证读操作能看到最新的 committed 数据，MVCC 快照读保证看到的数据是某一时刻的完整快照。

# Raft KV Store 幂等性改进方案

## 一、当前问题分析

### 1.1 幂等性定义

**幂等操作**：无论执行多少次，最终结果都相同。

对于 KV Store：
- `PUT key=value` 执行 1 次：key = value
- `PUT key=value` 执行 10 次：key = value（应该相同）

### 1.2 当前实现的问题

#### 问题代码

```java
// RaftKVService.java
public KVResponse put(String key, String value) {
    // ❌ 每次都生成新的 requestId
    String requestId = UUID.randomUUID().toString();
    
    // ❌ 没有检查是否已处理过这个请求
    KVTask task = KVTask.put(key, value, requestId);
    
    // ❌ 直接提交到 Raft
    node.apply(raftTask);
}
```

#### 问题场景

**场景 1：网络超时导致客户端重试**

```
时间线：
T1: 客户端 PUT name=Alice (requestId=uuid-1)
T2: Leader 写入日志，开始复制
T3: 复制成功，但响应慢（网络延迟）
T4: 客户端超时（3秒），没收到响应
T5: 客户端重试 PUT name=Alice (requestId=uuid-2)  ← 新的 requestId
T6: Leader 再次写入日志

结果：
- Raft 日志中有 2 条记录
- 状态机应用了 2 次
- 虽然最终结果相同（name=Alice）
- 但浪费资源，且无法区分是否重复
```

**场景 2：超时后数据状态不确定**

```
客户端发送 PUT，超时返回 "Write timeout"

客户端不知道：
- 操作是否成功？（可能成功但响应慢）
- 是否需要重试？
- 重试会不会产生重复数据？
```

**场景 3：Leader 切换**

```
T1: 客户端发送 PUT 到 Leader1
T2: Leader1 写入日志，还没复制完成就宕机
T3: Leader2 赢得选举
T4: 客户端重试 PUT 到 Leader2

问题：
- Leader2 的日志中没有这条记录
- 客户端重试是合理的
- 但无法区分是重试还是新请求
```

## 二、解决方案

### 方案 1：基于 RequestId 的去重机制（⭐⭐⭐⭐⭐ 推荐）

**核心思路**：
1. 客户端生成 requestId（不是服务端）
2. 服务端维护已处理请求的缓存
3. 重复请求直接返回缓存结果，不重复执行

#### 2.1.1 修改 KVTask

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class KVTask implements Serializable {
    private String op;
    private String key;
    private String value;
    private long timestamp;
    
    /**
     * 客户端生成的 requestId（用于幂等去重）
     * 必须由客户端生成，保证重试时 requestId 相同
     */
    private String requestId;
    
    public static KVTask put(String key, String value, String requestId) {
        return new KVTask(OP_PUT, key, value, System.currentTimeMillis(), requestId);
    }
}
```

#### 2.1.2 添加请求去重缓存

```java
// KVStoreStateMachine.java

/**
 * 已处理请求的缓存（用于幂等去重）
 * Key: requestId
 * Value: 操作结果
 */
private final Map<String, KVResponse> processedRequests = new ConcurrentHashMap<>();

/**
 * 请求缓存的最大大小（防止内存溢出）
 */
private static final int MAX_CACHE_SIZE = 10000;

@Override
public void onApply(Iterator iterator) {
    while (iterator.hasNext()) {
        long index = iterator.getIndex();
        try {
            ByteBuffer dataBuffer = iterator.getData();
            if (dataBuffer != null && dataBuffer.hasRemaining()) {
                byte[] data = new byte[dataBuffer.remaining()];
                dataBuffer.get(data);
                KVTask task = deserialize(data);
                if (task != null) {
                    // 幂等检查：如果 requestId 已处理，跳过执行
                    if (isDuplicateRequest(task.getRequestId())) {
                        LOG.debug("Duplicate request skipped: {}", task.getRequestId());
                    } else {
                        // 执行操作并缓存结果
                        applyTask(task);
                        cacheResult(task);
                    }
                    LOG.debug("Applied task at index {}: {} key={}", index, task.getOp(), task.getKey());
                }
            }
        } catch (Exception e) {
            LOG.error("Failed to apply task at index {}: {}", index, e.getMessage(), e);
        } finally {
            iterator.next();
        }
    }
    saveData();
}

/**
 * 检查是否是重复请求
 */
private boolean isDuplicateRequest(String requestId) {
    return processedRequests.containsKey(requestId);
}

/**
 * 缓存操作结果
 */
private void cacheResult(KVTask task) {
    // 限制缓存大小，防止内存溢出
    if (processedRequests.size() > MAX_CACHE_SIZE) {
        // 清除最早的缓存（简化实现）
        processedRequests.clear();
    }
    
    KVResponse response = KVResponse.success(
        task.getKey(), 
        task.getValue(), 
        task.getRequestId(), 
        getCurrentEndpoint()
    );
    processedRequests.put(task.getRequestId(), response);
}

/**
 * 获取已缓存的请求结果（用于重试）
 */
public KVResponse getCachedResult(String requestId) {
    return processedRequests.get(requestId);
}
```

#### 2.1.3 修改 RaftKVService

```java
// RaftKVService.java

/**
 * 向 Raft 集群提交一个 PUT 操作（支持幂等）
 * 
 * @param key 键
 * @param value 值
 * @param requestId 客户端生成的请求 ID（重试时必须相同）
 * @return 操作结果
 */
public KVResponse put(String key, String value, String requestId) {
    // 检查当前节点是否是 Leader
    if (!isLeader()) {
        return redirectToLeader(key, value);
    }

    // 检查是否是重复请求（快速失败）
    KVResponse cachedResult = stateMachine.getCachedResult(requestId);
    if (cachedResult != null) {
        LOG.info("Request already processed, returning cached result: {}", requestId);
        return cachedResult;
    }

    try {
        // 使用客户端提供的 requestId（不是服务端生成）
        KVTask task = KVTask.put(key, value, requestId);

        byte[] data = objectMapper.writeValueAsBytes(task);
        ByteBuffer dataBuffer = ByteBuffer.wrap(data);
        CountDownLatch latch = new CountDownLatch(1);
        AtomicReference<Status> statusRef = new AtomicReference<>();

        Task raftTask = new Task(dataBuffer, new com.alipay.sofa.jraft.Closure() {
            @Override
            public void run(Status status) {
                statusRef.set(status);
                latch.countDown();
            }
        });

        node.apply(raftTask);

        boolean success = latch.await(raftProperties.getWriteTimeout(), TimeUnit.MILLISECONDS);
        if (!success) {
            return KVResponse.failure("Write timeout", requestId);
        }

        Status status = statusRef.get();
        if (status.isOk()) {
            LOG.info("PUT successful: key={}, value={}, requestId={}", key, value, requestId);
            return KVResponse.success(key, value, requestId, getCurrentEndpoint());
        } else {
            return KVResponse.failure(status.getErrorMsg(), requestId);
        }
    } catch (Exception e) {
        LOG.error("PUT failed: key={}", key, e);
        return KVResponse.failure(e.getMessage(), requestId);
    }
}
```

#### 2.1.4 修改 Controller

```java
// KVController.java

@PutMapping("/{key}")
public ResponseEntity<KVResponse> put(
        @PathVariable String key,
        @RequestBody Map<String, Object> body) {

    String value = (String) body.get("value");
    if (value == null) {
        return ResponseEntity.badRequest()
                .body(KVResponse.failure("Missing 'value' in request body", null));
    }

    // 客户端提供 requestId（可选）
    String requestId = (String) body.get("requestId");
    if (requestId == null) {
        requestId = UUID.randomUUID().toString();
    }

    log.info("PUT request: key={}, value={}, requestId={}", key, value, requestId);
    KVResponse response = raftKVService.put(key, value, requestId);

    if (!response.isSuccess() && "NOT_LEADER".equals(response.getError())) {
        return ResponseEntity.status(301)
                .header("Location", "http://" + response.getLeaderEndpoint() + "/kv/" + key)
                .body(response);
    }

    return ResponseEntity.ok(response);
}
```

#### 2.1.5 客户端使用示例

```java
// 客户端代码
public class KVClient {
    
    private String generateRequestId(String key, String value) {
        // 基于业务逻辑生成 requestId
        // 相同业务操作生成相同 requestId
        return "put-" + key + "-" + System.currentTimeMillis();
    }
    
    public KVResponse putWithRetry(String key, String value, int maxRetries) {
        String requestId = generateRequestId(key, value);  // 生成 requestId
        
        for (int i = 0; i < maxRetries; i++) {
            try {
                KVResponse response = httpClient.put("/kv/" + key, 
                    Map.of("key", key, "value", value, "requestId", requestId));
                
                if (response.isSuccess()) {
                    return response;  // 成功
                }
                
                // 如果是重复请求，直接返回
                if ("DUPLICATE_REQUEST".equals(response.getError())) {
                    return response;
                }
                
                // 超时或其他错误，重试
                LOG.warn("PUT failed, retrying... attempt={}", i + 1);
                
            } catch (Exception e) {
                LOG.error("PUT exception, retrying...", e);
            }
        }
        
        return KVResponse.failure("Max retries exceeded", requestId);
    }
}
```

### 方案 2：基于业务键的幂等（简单场景）

**思路**：对于 PUT 操作，相同的 key 多次写入相同值是天然幂等的。

**优点**：实现简单
**缺点**：无法区分是否是重复请求，浪费资源

```java
// KVStoreStateMachine.java
private void applyTask(KVTask task) {
    synchronized (kvStore) {
        switch (task.getOp()) {
            case KVTask.OP_PUT:
                String currentValue = kvStore.get(task.getKey());
                
                // 如果值相同，认为是重复请求（可选优化）
                if (task.getValue().equals(currentValue)) {
                    LOG.debug("PUT skipped (same value): key={}", task.getKey());
                } else {
                    kvStore.put(task.getKey(), task.getValue());
                    LOG.info("PUT: key={}, value={}", task.getKey(), task.getValue());
                }
                break;
        }
    }
}
```

### 方案 3：CAS（Compare-And-Swap）机制

**思路**：PUT 时携带期望的旧值，只有旧值匹配才更新。

```java
public class KVTask {
    private String op;
    private String key;
    private String value;
    private String expectedValue;  // 期望的旧值（用于 CAS）
    private String requestId;
}

// 状态机应用
case KVTask.OP_PUT:
    String currentValue = kvStore.get(task.getKey());
    if (task.getExpectedValue() == null || 
        task.getExpectedValue().equals(currentValue)) {
        kvStore.put(task.getKey(), task.getValue());
    } else {
        LOG.warn("CAS failed: key={}, expected={}, actual={}", 
            task.getKey(), task.getExpectedValue(), currentValue);
    }
    break;
```

## 三、失败场景及客户端处理策略

### 3.1 超时处理

**客户端代码：**

```java
public KVResponse putWithTimeout(String key, String value) {
    String requestId = generateRequestId(key, value);
    
    try {
        KVResponse response = httpClient.put("/kv/" + key, 
            Map.of("value", value, "requestId", requestId),
            timeout = 3000);
        return response;
    } catch (TimeoutException e) {
        // 超时后，客户端不知道是否成功
        // 策略：重试，使用相同的 requestId
        LOG.warn("PUT timeout, retrying with same requestId");
        return putWithRetry(key, value, requestId, 3);
    }
}
```

### 3.2 Leader 切换处理

**客户端代码：**

```java
public KVResponse putWithLeaderRedirect(String key, String value, String requestId) {
    String currentLeader = leaderEndpoint;  // 缓存的 Leader 地址
    
    for (int i = 0; i < 3; i++) {
        try {
            KVResponse response = httpClient.put(
                "http://" + currentLeader + "/kv/" + key,
                Map.of("value", value, "requestId", requestId)
            );
            
            if (response.isSuccess()) {
                return response;
            }
            
            // 如果不是 Leader，获取新的 Leader 地址
            if ("NOT_LEADER".equals(response.getError())) {
                currentLeader = response.getLeaderEndpoint();
                LOG.info("Redirecting to new Leader: {}", currentLeader);
                continue;
            }
            
            return response;
            
        } catch (Exception e) {
            LOG.error("PUT failed, retrying...", e);
        }
    }
    
    return KVResponse.failure("Max retries exceeded", requestId);
}
```

### 3.3 网络分区处理

**检测策略：**
1. 连续多次超时
2. 健康检查失败

**处理策略：**
1. 尝试其他节点
2. 等待网络恢复
3. 返回错误给客户端

```java
public KVResponse putWithPartitionTolerance(String key, String value) {
    String requestId = generateRequestId(key, value);
    List<String> peers = Arrays.asList(
        "http://localhost:9081",
        "http://localhost:9082",
        "http://localhost:9083"
    );
    
    for (String peer : peers) {
        try {
            KVResponse response = httpClient.put(peer + "/kv/" + key,
                Map.of("value", value, "requestId", requestId));
            
            if (response.isSuccess()) {
                return response;
            }
            
            // 如果是 NOT_LEADER，获取 Leader 地址
            if ("NOT_LEADER".equals(response.getError())) {
                String leader = response.getLeaderEndpoint();
                if (leader != null) {
                    peer = "http://" + leader;
                }
            }
            
        } catch (Exception e) {
            LOG.warn("Peer {} unavailable, trying next...", peer);
        }
    }
    
    return KVResponse.failure("All peers unavailable", requestId);
}
```

## 四、最佳实践总结

### 4.1 服务端

1. ✅ **维护已处理请求缓存**：基于 requestId 去重
2. ✅ **返回明确的错误码**：区分超时、NOT_LEADER、重复请求等
3. ✅ **限制缓存大小**：防止内存溢出（LRU 淘汰）
4. ✅ **持久化 requestId**：重启后仍能去重（可选）

### 4.2 客户端

1. ✅ **生成幂等 requestId**：基于业务逻辑（如：订单号、用户ID+时间戳）
2. ✅ **重试时使用相同 requestId**：保证幂等性
3. ✅ **处理重定向**：自动跟随 Leader 切换
4. ✅ **设置合理的超时和重试次数**：避免无限重试
5. ✅ **实现指数退避**：避免雪崩

### 4.3 RequestId 生成策略

```java
// 策略 1：UUID（简单，但重试时不同）
String requestId = UUID.randomUUID().toString();

// 策略 2：业务键 + 时间戳（推荐）

// 策略 3：业务键 + 版本号（最严格）
String requestId = "put-" + key + "-v" + version;

// 策略 4：客户端 ID + 序列号（分布式友好）
String requestId = clientId + "-" + sequence.incrementAndGet();
```

## 六、总结

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| RequestId 去重 | 严格幂等，可重试 | 需要客户端配合 | 生产环境（推荐） |
| 业务键幂等 | 实现简单 | 无法区分重复请求 | 简单场景 |
| CAS 机制 | 精确控制 | 需要知道旧值 | 并发更新场景 |

**推荐方案**：RequestId 去重机制
- 严格的幂等保证
- 支持安全的重试
- 符合业界最佳实践
